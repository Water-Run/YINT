# `yint`: YINT Is Not TLS

`yint`是一个简单的对称加密协议, 用于在老旧的不可改造的HTTP信道上安全的接口调用和数据传输.  
`yint`项目在[GitHub](https://github.com/Water-Run/YINT)上以`LGPL`协议开源, 提供:  

- 协议内容  
- 核心实现. 包括二进制版本  
- 现代`Python`库(`libs/py`, Python 3.12+)  
- 现代`PHP`库(`libs/php`, PHP 8.3+)  
- 现代`Rust`库(`libs/rust`, Rust edition 2024)  
- `Lua`库(`libs/lua`, Lua 5.1+)  
- Shell 库 / 命令入口: `bat`、`ps5`、`ps`、`bash`、`fish`、`zsh`  
- 兼容老环境的`PHP`库(`libs/php52`, PHP 5.2)  

## 基本  

`yint`协议在客户端和服务端之间预共享一个`master_key`, 通过它派生出加密密钥`K_enc`和签名密钥`K_mac`. 在此基础上, 协议同时提供以下基本能力:  

- **机密性**: 请求与响应的`body`始终使用`AES-256-CBC`加密, 协议层不存在明文模式.  
- **完整性**: `body`使用`HMAC-SHA256`保护, 采用`Encrypt-then-MAC`结构.  
- **真实性**: 整个请求(`METHOD`/`URI`/`BODY`)由`HMAC-SHA256`签名, 第三方无法伪造或篡改.  
- **防重放**: 通过`timestamp`+`nonce`组合, 服务端拒绝过期或重复的请求.  
- **响应防伪**: 服务端响应同样附带签名, 客户端可校验响应未被中间人篡改或伪造.  

整个协议设计**极简**:  

- 单一对称密钥, 不存在密钥协商、证书链、握手过程.  
- 仅 3 个固定请求头(`X-Yint-Timestamp` / `X-Yint-Nonce` / `X-Yint-Sign`), 三者全部强制.  
- 无版本号、无多密钥、无密钥轮换. 需要更换密钥时双端同时停机替换.  
- 不提供分级加密、不提供`KeyId`、不提供压缩/流式/分块、不提供身份概念. 协议只做一件事: 让一次`HTTP`往返变得安全.  
- 库只负责"构造请求 / 校验请求 / 构造响应 / 解析响应"四件事, **不接管 HTTP 监听**, 对原有接口的侵入度极低.  

服务端仅有一个可调参数: 时间窗口大小`time_window`(默认 300 秒). 该参数同时决定时间戳校验范围与`nonce`在服务端的存活时长——任何在窗口外的请求都会被时间戳校验拒绝, 不会进入查重表; 窗口内的`nonce`记录在服务端进程内存中, 出窗后自动丢弃. **不支持跨进程/跨机器共享**.  

## 安全性  

`yint`在它所定位的场景中提供如下保证:  

- 攻击者在不持有`master_key`的前提下, **无法**伪造合法请求、**无法**伪造合法响应、**无法**解密任何被截获的`body`.  
- 已截获的合法请求, 在`time_window`之外重放会被时间戳校验拒绝; 在`time_window`之内重放会被`nonce`查重拒绝.  
- 中间人对密文`body`的任何位翻转都会被`HMAC`校验拦截, 不会进入业务逻辑.  

`yint`**不**提供以下保证:  

- **不提供前向安全性**. `master_key`一旦泄露, 历史流量可被解密. 这是任何长期对称密钥协议的固有限制.  
- **不隐藏元数据**. `URL`、`PATH`、`QUERY`、请求方法、请求头、`body`长度等信息在`HTTP`层依然可见.  
- **不抵御侧信道攻击**. 由于必须兼容`PHP 5.2`, 协议实现使用纯`PHP`/纯`Python`完成`AES`运算, 在面对计时攻击、缓存攻击等侧信道威胁时**不具备**与`OpenSSL`同等的防护. 如果你的威胁模型包含本地侧信道攻击者, 请勿使用`yint`.  
- **不抵御端点失陷**. 客户端或服务端任一方被攻陷, `master_key`即泄露, 协议全面失效.  
- **不替代认证授权**. `yint`只证明"对端持有`master_key`", 不区分用户身份. 业务层的鉴权(登录态 / 权限 / 配额)需另行设计.
